## Project Overview
A living ML-driven trade signal system for small-cap stocks, designed to expand into mid-caps, precious metals, and crypto over time. The system generates trade profiles (entry, target, stop-loss, confidence score, expected hold time) and continuously re-evaluates open signals on a scheduled basis throughout the trading week. Outcome tracking is fully autonomous — no manual labeling. The project is treated as a true software engineering project: ship fast, iterate often, and keep the codebase simple and maintainable.

## Repo Structure
The project is organized around five core domains: (1) Data — ingestion, normalization, and storage of market data, starting with Yahoo Finance but architected to swap providers with minimal friction; (2) Modeling — a simple, reproducible ML model as the foundation, versioned and built to be improved incrementally; (3) Signals — generation of trade profiles and a re-evaluation loop that refreshes open signals multiple times per day; (4) Backtesting — historical validation of models and retrospective scoring of closed signals to identify where entries or exits could have been improved; (5) Pipeline — a DAG-based scheduler that orchestrates data ingestion, signal generation, re-evaluation, and outcome tracking at defined intervals. A lightweight data store persists signals, outcomes, and model versions across runs.

## Tech Stack & Conventions
Python is the primary language. UV is the required package manager — no pip or poetry. Yahoo Finance is the initial data source, with the architecture designed to accommodate paid providers (e.g., Polygon.io, Tiingo) as needed. A DAG-based pipeline framework (e.g., Prefect or similar lightweight option) handles scheduling. A lightweight database (e.g., SQLite to start, upgradeable) stores signals and outcomes. ML tooling is to be selected based on modern best practices and what has proven effective for financial time-series tasks.

## Out-of-Scope Constraints
1. Simple is better than complex. Complex is better than complicated. 2. Documentation must be human-readable — avoid jargon-heavy or overly long AI-generated prose. 3. UV is mandatory as the package manager. 4. Solo developer — avoid over-engineering the team or process overhead. 5. No manual data labeling — all outcome tracking and feedback loops must be fully automated. 6. Budget is open for quality data sources. 7. Timeline is rolling — prioritize working software over perfect software. 8. The first model must be intentionally minimal and reproducible before any complexity is added.
